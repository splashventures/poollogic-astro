# Plan: Schema, GEO, and PageSpeed optimizations for PoolLogic Astro site

## Assumptions
- Astro 5 site deployed on Vercel, static output, 224 pages
- Existing schema infrastructure in src/lib/schema/ with modular generators
- Reviews page already has individual Review schema embedded in LocalBusiness schema
- Blog posts already have dateModified in their data model
- Hero image at /images/pool/hero-pool.webp on homepage

## Acceptance Criteria
1. HowTo JSON-LD appears on all 16 service pages with correct steps from getServiceSteps()
2. Speakable schema appears on city and service pages with .definition-paragraph selector
3. GeoShape polygon added to main LocalBusiness schema covering San Diego County
4. Reviews page has individual Review schemas (already present, verify)
5. "Last Updated" date visible on blog posts, city pages, and service pages
6. City pages open with a definition-first paragraph with class="definition-paragraph"
7. Direct-answer blocks appear after main H2 headings on city/service pages
8. Quantified statistics (pools served, distance, years) prominently displayed on city pages
9. Astro ViewTransitions enabled in BaseLayout
10. Hero image has fetchpriority="high", preload link in head
11. All images have explicit width/height; non-hero images use loading="lazy"
12. Font loading optimized with font-display: swap
13. `astro build` succeeds with all 224 pages
14. PR created against main branch

## Decision Log
- Used font-display: swap override in globals.css rather than patching @fontsource source files
- Used compressHTML: true in astro config for automatic HTML minification
- Added preloadImage prop to BaseLayout so only homepage gets the hero image preload
- Used polygon format for GeoShape using all 22 city coordinates from cities.ts

## Surprises & Discoveries
- Reviews page already has individual Review schema objects nested inside a LocalBusiness schema
- Article schema already includes dateModified
- All 22 city coordinates available in cities.ts for GeoShape polygon
- Vite type-only import warnings are expected and harmless (known issue from initial migration)
