# Plan: Implement remaining framework evaluation nice-to-haves

## Assumptions
- Astro 5.7 project with static output, deployed to Vercel
- All content is in TypeScript data files (no CMS)
- @astrojs/sitemap is already installed but has no custom configuration
- Images are in `public/images/` using standard `<img>` tags (21 occurrences across 11 files)
- Brand data (values, milestones, team, whyChooseUs) is hardcoded in about/index.astro frontmatter
- Section ordering is implicit in page templates, not data-driven
- sharp image service already configured at quality 80

## Acceptance Criteria
1. Brand data (values, milestones, teamMembers, whyChooseUs) lives in `src/data/brand.ts`, exported with TypeScript types, and `/about/` page imports from there instead of defining inline
2. `src/data/page-layouts.ts` exists with section ordering for homepage, about, service, and city page types, exported as typed arrays
3. Sitemap config in `astro.config.mjs` uses `serialize` callback to assign priority tiers: homepage 1.0, service pages 0.9, city pages 0.8, about 0.7, blog posts 0.6, legal/utility pages 0.3
4. All `<img>` tags in components and pages are replaced with Astro's `<Image>` component using images imported from `src/assets/images/`; `npm run build` succeeds with no errors
5. Running `npm run build` completes successfully with all changes applied

## Approach
1. Create `src/data/brand.ts` with values, milestones, teamMembers, whyChooseUs arrays
2. Update about/index.astro to import from brand.ts
3. Create `src/data/page-layouts.ts` defining section order per page type
4. Add sitemap priority tiers via serialize callback in astro.config.mjs
5. Move images from public/ to src/assets/images/ and migrate <img> to <Image>
6. Build and fix any issues
7. Commit and create PR

- Used ternary fallback pattern: `<Image>` when asset exists in src/assets, `<img>` otherwise
- Copied all images from public/ to src/assets/ (logos, pool, team, blog) for Astro optimization
- Kept public/ images intact since they may be referenced by external systems or schema URLs
- Used `import.meta.glob` with eager loading for team headshots on about page since paths come from brand.ts data
- Organization schema numberOfEmployees now uses `teamMembers.length` instead of hardcoded 12
